# Kubernetes Configuration

## ConfigMaps

### Motivation
- Decouple environment-specific configuration from container images
- Store non-confidential configuration data in key-value pairs
- Applications can be easily ported across environments

### ConfigMap Object
- API object with `data` and `binaryData` fields (no `spec`)
- `data`: UTF-8 strings as key-value pairs
- `binaryData`: Binary data as base64-encoded strings
- Keys must be alphanumeric, `-`, `_`, or `.`
- Keys in `data` must not overlap with `binaryData`
- Name must be a valid DNS subdomain name
- `immutable: true` creates immutable ConfigMap (v1.19+)

### Using ConfigMaps

#### As Files in a Pod (Volume)
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-pod
spec:
  containers:
  - name: test
    image: busybox
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config
  volumes:
  - name: config-volume
    configMap:
      name: game-config
```

#### As Environment Variables
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: env-configmap
spec:
  containers:
  - name: test
    image: busybox
    env:
    - name: GAME_MODE
      valueFrom:
        configMapKeyRef:
          name: game-config
          key: game.mode
    envFrom:
    - configMapRef:
        name: game-config
```

### Immutable ConfigMaps
- Set `immutable: true` to prevent modifications
- Improves performance (kubelet doesn't poll for changes)
- To change, create a new ConfigMap and update the reference

## Secrets

### Overview
- Object containing sensitive data (passwords, tokens, keys)
- Prevents including confidential data in application code
- Smaller than 1MB by default
- Base64-encoded (not encrypted by default; enable encryption at rest)

### Uses for Secrets
- As environment variables in a Pod
- As files mounted in a Pod (Secret volume)
- For image pull credentials (imagePullSecrets)
- Visible to specific containers only in a multi-container Pod

### Types of Secret

#### Opaque Secrets (default)
- Arbitrary user-defined data
- Created with `kubectl create secret generic`

#### ServiceAccount token Secrets
- Holds token for ServiceAccount
- Auto-generated by controller

#### Docker config Secrets
- `.dockerconfigjson` for registry authentication
- `kubernetes.io/dockercfg` (legacy) or `kubernetes.io/dockerconfigjson`

#### Basic authentication Secret
- `kubernetes.io/basic-auth` type
- Contains `username` and `password` keys

#### SSH authentication Secret
- `kubernetes.io/ssh-auth` type
- Contains `ssh-privatekey` key

#### TLS Secret
- `kubernetes.io/tls` type
- Contains `tls.crt` and `tls.key` keys
- Created with `kubectl create secret tls`

#### Bootstrap token Secret
- `bootstrap.kubernetes.io/token` type
- Used for joining nodes to cluster

### Creating a Secret
```bash
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password=S3cur3
```

### Using Secrets

#### As Files from a Pod
```yaml
volumes:
- name: secret-volume
  secret:
    secretName: my-secret
```

#### As Environment Variables
```yaml
env:
- name: SECRET_USERNAME
  valueFrom:
    secretKeyRef:
      name: my-secret
      key: username
```

#### Container Image Pull Secrets
```yaml
spec:
  imagePullSecrets:
  - name: regcred
```

### Immutable Secrets
- Set `immutable: true` to prevent modifications
- Improves performance and security
- To change, create a new Secret and update references

### Information Security
- Enable encryption at rest for Secret data in etcd
- Configure least-privilege access via RBAC
- Avoid writing Secret data to logs
- Consider external secret management (Vault, cloud KMS) for sensitive data
- Secrets in environment variables can be exposed via process inspection

## Resource Management

### Resource Requests and Limits
- **Requests**: Guaranteed minimum resources; used for scheduling
- **Limits**: Maximum resources; enforced at runtime
- CPU: measured in cores (1 core = 1000m millicores)
- Memory: measured in bytes (Ki, Mi, Gi, Ti)

```yaml
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        cpu: 250m
        memory: 64Mi
      limits:
        cpu: 500m
        memory: 128Mi
```

### CPU Units
- 1 CPU = 1 AWS vCPU / 1 GCP Core / 1 Azure vCore / 1 Hyperthread
- Expressed as: `1`, `0.5`, or `500m` (millicores)

### Memory Units
- Bytes (default), Ki (1024 bytes), Mi, Gi, Ti, Pi, Ei
- Or: K, M, G, T, P, E (powers of 1000)

### Enforcement
- CPU limits: enforced via CPU throttling (CFS quota)
- Memory limits: enforced via OOM killer
- Requests: used by kube-scheduler for node placement

### QoS Classes
- **Guaranteed**: requests == limits for all resources (highest priority)
- **Burstable**: requests < limits (at least one resource)
- **BestEffort**: no requests or limits (lowest priority)

## kubeconfig

### Overview
- Configuration file for cluster access (default: `$HOME/.kube/config`)
- Contains clusters, users, and contexts
- Can be specified via `KUBECONFIG` env var or `--kubeconfig` flag

### Structure
```yaml
apiVersion: v1
kind: Config
clusters:
- name: my-cluster
  cluster:
    server: https://my-cluster-api:6443
    certificate-authority-data: <base64-ca-cert>
users:
- name: my-user
  user:
    client-certificate-data: <base64-cert>
    client-key-data: <base64-key>
contexts:
- name: my-context
  context:
    cluster: my-cluster
    user: my-user
    namespace: default
current-context: my-context
```

### Managing Contexts
```bash
kubectl config get-contexts              # List all contexts
kubectl config current-context           # Show current context
kubectl config use-context my-context    # Switch context
kubectl config set-context --current --namespace=my-ns  # Change namespace
```
